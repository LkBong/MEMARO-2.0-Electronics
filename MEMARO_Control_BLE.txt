⏺ Recommendation: neither — keep it to @MainActor ObservableObject with 
  @Published.

  Here's why the "pick one" framing is a false dilemma:

  @Published is Combine under the hood, and SwiftUI's data flow already binds to
   it. Adding PassthroughSubject chains on top creates a second reactive layer
  with zero benefit. Wrapping every delegate callback into an AsyncStream adds
  boilerplate for a problem that doesn't need it.

  The BLE use case here is simple:

  ┌────────────────────────────┬────────────────────────────────────────────┐
  │         Event type         │                  Pattern                   │
  ├────────────────────────────┼────────────────────────────────────────────┤
  │ Continuous feedback (RPM,  │ Update @Published directly in delegate     │
  │ current every ~50 ms)      │ callback                                   │
  ├────────────────────────────┼────────────────────────────────────────────┤
  │ State changes (connected,  │ Update @Published directly                 │
  │ disconnected, scanning)    │                                            │
  ├────────────────────────────┼────────────────────────────────────────────┤
  │ One-shot sequences (scan → │ async/await + withCheckedContinuation only │
  │  pick → connect →          │  if it reads cleaner than a state machine  │
  │ discover)                  │                                            │
  └────────────────────────────┴────────────────────────────────────────────┘

  Concrete approach:

  @MainActor
  final class BLEManager: NSObject, ObservableObject {
      @Published var rpm: Int = 0
      @Published var currentA: Double = 0
      @Published var isConnected: Bool = false
      @Published var peripherals: [CBPeripheral] = []

      // Delegate callback — just assign, SwiftUI reacts automatically
      func peripheral(_ p: CBPeripheral, didUpdateValueFor characteristic:
  CBCharacteristic, error: Error?) {
          guard let data = characteristic.value, data.count == 4 else { return }
          rpm      = Int(data[0]) << 8 | Int(data[1])
          currentA = Double(Int(data[2]) << 8 | Int(data[3])) / 1000.0
      }
  }

  No .sink, no AnyCancellable bag, no AsyncStream. The @MainActor annotation
  ensures thread safety — CoreBluetooth calls delegates on its own queue, and
  @MainActor routes the work to the main thread automatically.

  async/await would add value if you were chaining scan → connect → discover as
  a structured sequence, but given the device picker UX (user taps a peripheral,
   then connect happens), the delegate state machine is already the natural fit.
